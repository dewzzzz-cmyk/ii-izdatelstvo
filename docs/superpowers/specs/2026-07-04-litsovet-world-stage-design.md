# Литсовет — стадия «Мир» (проактивный worldbuilding) — Design Spec

**Дата:** 2026-07-04
**Статус:** утверждён к планированию
**Тип:** новая опциональная стадия в существующем пайплайне «Литсовет»

---

## 1. Проблема

Сейчас факты мира книги появляются **реактивно**: «архивариус» вытаскивает их из уже
написанного текста (см. `summarizer.js`, `series.js`), а Библия (`state.bible[]`)
заполняется вручную автором или расширяется точечно кнопкой «✨» в панели «Память».
Для жанров с придуманным сеттингом (фэнтези, фантастика, ромфант, ЛитРПГ, мистика/
ужасы, альтернативная история) это создаёт риск: магия/география/фракции/точка
развилки не зафиксированы **до** того, как Архитектор строит структуру книги — мир и
сюжет могут разъехаться, и это вскрывается только через много глав.

## 2. Цели и не-цели

### Цели
- Проактивный шаг «Мир»: до Структуры автор (при помощи ИИ) фиксирует ключевые факты
  придуманного сеттинга, они сразу становятся частью канона (Библии) и видны
  Архитектору при построении скелета.
- Жанрово-осознанный охват: полноценный «Мир» для жанров с придуманным сеттингом,
  не мешает и не показывается по умолчанию для реалистичных (детектив, драма).
- Канал остаётся **открытым**, а не одноразовым гейтом: Архитектор может запросить
  недостающий факт уже во время Структуры; автор может дополнить мир после того как
  структура построена (с предупреждением, не блокировкой).
- Новый жанр «Альтернативная история» с осознанной поддержкой точки развилки (POD).
- Опциональная генерация карты мира как референса уже на этой стадии, использующая
  существующую инфраструктуру иллюстраций.

### Не-цели (YAGNI)
- Отдельная структура данных `state.world` — переиспользуем `state.bible[]`.
- Вычитка/структурная валидация карты мира как источника истины для консистентности
  (карта — картинка, её не читают агенты; текстовые факты Библии остаются якорем).
- Обязательные поля/жёсткая форма для каждой категории — генерация и ревью остаются
  гибкими, автор может пропустить любую категорию.
- Полноценный редактор карт (легенды, слои, интерактивность) — просто сгенерированное
  изображение по промпту, как обложка/иллюстрация сцены.

## 3. Место в пайплайне и включение стадии

Новая стадия `world` (`{ id:'world', label:'Мир' }`) в `STAGES` (`ui/app.js`),
физически между `concept` и `structure`. Видимость управляется `project.useWorld`
(bool), по прямой аналогии с существующим `project.useVoice`/стадией «Голос»
(`ui/app.js:62`, `renderRail`).

**Авто-включение по жанру** (в отличие от `useVoice`, который включается только
вручную): при выборе жанра в Концепции (`genreSel.onchange`, `ui/stages.js`)
`useWorld` авто-проставляется `true` для: фэнтези, фантастика (НФ), ромфант, ЛитРПГ,
мистика/ужасы, альтернативная история (новый жанр, см. §7). Для остальных — `false`,
но чекбокс всегда кликабелен вручную (детектив с придуманным городом и т.п.).

**Важно (правка по итогам экспертного ревью):** авто-включение — решение системы,
не автора, поэтому должно быть **видимым**, а не только тихим флагом. Решение по
итогам спек-ревью (упрощено под YAGNI — без состояния «показать/скрыть/погасить»):
статичная надпись рядом с чекбоксом «Мир» в блоке «▾ Дополнительные настройки»,
видимая всегда, когда `useWorld===true`: «Включено автоматически для жанра
«{genre}» — можно выключить». Никакой отдельной «разовой» подсказки с собственным
жизненным циклом не заводим — чекбокс и так в том же месте, что и `useVoice`.

Кнопка «Дальше» на Концепции ведёт на `world`, если `useWorld`, иначе на `voice`
(если включён) или `structure` — та же цепочка условий, что уже реализована
для `useVoice` (`ui/stages.js:313`).

## 4. Данные: переиспользование Библии

Одобренные факты мира пишутся как обычные записи `state.bible[]`:
`{ keys, text, source:'world' }`. Поле `source` — новое, но **не требует миграции**:
у старых записей оно `undefined`, любая проверка `b.source==='world'` для них просто
`false` — безопасно (подтверждено чтением `bible.js`/`state.js migrate()`).

`project.useWorld:false` добавляется в `defaultState()` — тоже без ручной миграции:
`migrate()` уже мержит `project` через `Object.assign({}, d.project, s.project)`
(`state.js`), новый дефолт подхватится автоматически, как было с `useVoice`.

TF-IDF-поиск (`bibleMatches`/`bibleForPrompt`, `bible.js`) не меняется вообще —
world-факты для Прозаика/Стражей проходят через существующий top-5 семантический
поиск как обычные записи канона. Рекомендация авторам world-агента: держать факты
такими же конкретными, как остальные записи Библии (не «в этом мире есть магия», а
«боевая магия истощает год жизни за каждое применение») — иначе общие/шумные
формулировки могут конкурировать за top-5 с более релевантными сценическими фактами
из-за низкого порога `cosine>0.08`.

## 5. Агент «Мир»: генерация фактов

Новый файл `src/world.js` (по образцу `src/illustrations.js` — суггест-агент на
существующем текстовом LLM, без отдельного API-ключа/денег).

### 5.1 Затравка — точечные подсказки, не одно поле
Вместо одного свободного текстового поля — общее необязательное поле «идея мира»
(как `idea`/`synopsis` в Концепции) + 3-5 необязательных однострочных точечных
подсказок с плейсхолдерами, привязанных к уже известным жанру/синопсису:
- «Что герой не может получить/сделать без магии/технологии? (это и есть её главное
  ограничение)»
- «Какая фракция/сила антагонистична герою и почему?»
- (для альт-истории — см. §7 — точка развилки обязательна, не в этом списке)

Всё необязательно — автор может нажать «Предложить мир» вообще без ввода, агент
использует жанр+синопсис из Концепции.

### 5.2 Категории (genre-aware)

Хранится как константа в `world.js`, аналогично тому, как `genres.js` хранит
жанровые ветки — не свободная генерация набора категорий агентом, а
детерминированный список на стороне кода, который передаётся агенту как часть
промпта:

```js
const BASE_CATEGORIES = ['география', 'история', 'фракции', 'культура'];
// добавляется/подменяется по жанру:
// − 'магия/технология' — добавляется, если жанр ∈ {фэнтези, фантастика, ромфант,
//   мистика/ужасы, другой (если автор включил useWorld вручную — см. ниже)}
// − 'магия/технология' → заменяется на 'система' для ЛитРПГ
// − для жанров вне авто-списка (useWorld включён вручную, напр. детектив) —
//   используется полный BASE_CATEGORIES без магии/технологии/системы, если
//   автор явно не попросил её через затравку
```

- **География** — места действия, расстояния, климат, что где находится.
- **История/предыстория** — ключевые прошлые события, формирующие настоящее.
  Для «альтернативная история» — обязательно включает точку развилки (§7).
- **Магия/технология** *(добавляется только для жанров из списка выше)* — как
  работает, и **явно** — что она НЕ может делать / чего стоит (агент прямо
  инструктируется спрашивать про ограничения наравне с позитивными фактами —
  иначе Прозаик тихо придумывает deus ex machina, а Стражи не могут это
  поймать, т.к. в Библии нет отрицательных фактов). Для ЛитРПГ эта категория
  заменяется на **«Система»** (правила прогрессии: уровни, статы, классы, есть
  ли пермасмерть).
- **Фракции/политика** — силы в конфликте, их цели.
- **Повседневность/культура** — включает явно соц. нормы, статус/обращение,
  религию/мировоззрение, не только быт (еда/одежда) — чтобы авторы не забывали
  именно эту часть, которая часто двигает драму (мезальянс, честь, табу).

### 5.3 Формат вывода
JSON-массив фактов: `{ category, keys, text }`. Валидация как у других агентов
(`extractJSON` + ретрай на невалидный JSON, паттерн `architect-book.js`).

## 6. UI ревью и одобрение

Новый файл `src/ui/world.js` (по образцу `src/ui/illustrations.js`).

- Кандидаты группируются по категории в **сворачиваемые секции** — переиспользуем
  паттерн, только что реализованный в панели «Память» (свёрнуто/развёрнуто по клику
  на заголовок секции), а не изобретаем новый. По умолчанию — развёрнуто.
- Внутри секции — карточки как в `illustrations.js` `renderCandidates`: чекбокс +
  текст факта (инлайн-редактируемый) + возможность удалить.
- Кнопка «+ свой факт» — добавить вручную в любую категорию (как `bibleAdd` в
  `ui/memory.js`).
- «Сохранить в канон» — одобренные факты уходят в `state.bible[]` с `source:'world'`,
  `rebuildBibleVecs()` пересчитывает векторы.

## 7. Не одноразовый гейт: открытый канал + устаревание

**Правка по итогам экспертного ревью (главная):** жёсткая последовательность «сначала
весь мир, потом вся структура» без обратной связи признана рискованной — worldbuilding
и сюжет на практике итеративны, потребности сюжета часто диктуют, какой факт мира
нужен, а не наоборот.

- **На стадии Структуры:** если Архитектор в процессе построения/улучшения скелета
  обнаруживает необходимость факта, которого нет в Библии, он может предложить
  дополнение канона тем же UI ревью, что и на стадии «Мир» (не блокирует генерацию
  скелета — предложение появляется рядом с результатом, автор одобряет/игнорирует).
- **Обратный случай:** если автор добавляет новые world-факты в Библию **после** того,
  как `state.structure` уже создан, проставляется независимый флаг
  `state.structureStale = true` (устанавливается в точке добавления world-факта в
  `ui/world.js`, не связан с оценкой структуры). На стадии «Структура» это новый,
  отдельный блок рендера в `renderStructure` — **визуально** похожий на существующий
  `didDrop`-баннер (`ui/stages.js:596-628`, рамка `var(--err)`), но НЕ переиспользующий
  его код или состояние (`didDrop` = `scoreDelta<0` между итерациями оценки — это
  другая величина). Текст: «Добавлены факты мира после построения структуры —
  возможно, стоит перестроить». Флаг сбрасывается при следующей (пере)генерации
  скелета. Не блокирует работу, просто информирует.

## 8. Архитектор: интеграция world-фактов

`bookArchitectMessages()` (`architect-book.js`) сейчас не читает `state.bible` вообще —
это новый код, не просто «добавить строку».

**Решение (правка по итогам ревью):** НЕ передавать «все» world-факты сырым списком.
Переиспользовать существующий `bibleForPrompt(bible.filter(b=>b.source==='world'),
synopsis, 15)` — top-15 через уже готовую TF-IDF-инфраструктуру вместо ручного отбора.
Это даёт естественный кап без нового кода валидации и не создаёт конкуренции за top-5
слотов у сценового Bible-поиска (у Архитектора свой, независимый K=15).

Блок вставляется в `user`-сообщение `bookArchitectMessages()` сразу после
`genreBeatsNote(p.genre)`, и остаётся **стабильным** между итерациями «Улучшить»
(`opts.previousSkeleton`/`opts.hint`) — не пересчитывается заново на каждой итерации
теми же входными данными, чтобы не вносить шум в реконсиляцию с `prevEval`.

## 9. Карта мира

Переиспользует существующий `src/imagegen.js` (`generateImage()`) — **не** новая
подсистема для самой генерации. Но **не** проходит через
`suggestIllustrations()`/`illustrationSuggestMessages()` (`illustrations.js`): эти
функции жёстко требуют `doneScenesOrdered(state)` (бросают ошибку «Нужна хотя бы
одна законченная сцена») — на стадии «Мир» сцен ещё физически не существует.

Поэтому промпт карты строится отдельной функцией `mapPromptFor(state)` в
`src/world.js`, не зависящей от написанных сцен — только от world-фактов категории
«География» (`state.bible.filter(b=>b.source==='world' && b.category==='география')`)
и жанра/эпохи из `project`. Специфичный для карты стиль (вид сверху, подписанные
локации, картографический жанр) — отдельный системный промпт, не переиспользующий
`illustrationSuggestMessages`. Из существующего потока переиспользуется **только**
низкоуровневый вызов `generateImage()` и запись результата в `state.illustrations.items`
— не весь suggest/candidates flow.

- Кнопка «🗺 Сгенерировать карту (референс)» на стадии «Мир», видна когда есть ≥2-3
  факта категории «География». Явная оценка стоимости перед кликом (как в
  Иллюстрациях) — деньги тратятся только по клику.
- Результат пишется в **тот же** `state.illustrations.items` с `type:'map'` — единый
  источник правды, не отдельное хранилище. Превью показывается сразу на стадии «Мир»
  (не только «идите смотреть в Иллюстрации») + подпись-ссылка «Также доступно в
  Иллюстрациях →».
- В разделе «Иллюстрации» карта видна в общей галерее как обычный item, с
  возможностью перегенерировать (если мир дополнился по ходу книги).

## 10. Новый жанр: Альтернативная история

Добавляется в `GENRES` (`genres.js`): `{ v:'альтернативная история',
label:'Альтернативная история', words: 95000 }`. Отдельная запись, не подпункт
«исторического романа» — читательские ожидания другие (не точность эпохи, а
последовательность альтернативных следствий).

- **`genreBeatsNote`:** точка развилки должна быть узнаваема с первых глав через то,
  как выглядит альтернативная «норма» (не через лекцию-экспозицию); причинно-
  следственная цепочка от развилки к текущему состоянию мира должна быть
  прослеживаема; кульминация часто завязана на угрозу «исправления» истории или на
  цену, которую развилка потребовала от героя/мира.
- **Категория «История» в стадии «Мир» для этого жанра — обязательная (не
  пропускаемая молча):** явная точка развилки — конкретное историческое событие + год
  + что пошло иначе + 2-3 конкретных следствия (технологические/политические/
  культурные), логически из неё вытекающих. Не хардблок (см. §7): если POD не задан,
  мягкое предупреждение «для альт-истории точка развилки — основа жанра, вы её не
  указали» с возможностью продолжить без неё.
- **Паттерн матчинга жанра:** `g.includes('альтернативн')` (где `g` — жанр в
  нижнем регистре, тот же паттерн что и у остальных веток `genres.js`). Проверено:
  не пересекается по подстроке с `'исторический роман'` (не содержит «альтернативн»)
  и не перехватывается им (порядок веток в `genres.js`/`genreBeatsNote` и др. в
  любом случае должен матчить «альтернативн» отдельной веткой, до общей ветки
  «истори», если такая появится в будущем).
- **`genreJudgeNote`:** Стражи/Оценщик не штрафуют кажущиеся анахронизмы, если они —
  прямое следствие заявленной точки развилки (сдвинутая на 50 лет раньше
  промышленная революция → продвинутая техника в 1850-м не ошибка). Противоречия
  самой заявленной причинно-следственной цепочке — обычная проверка континуити через
  Библию, ничего нового не требуется.

## 12. Настройки Книжного архитектора

Отдельное, но связанное дополнение (те же файлы: `architect-book.js`, Концепция) —
сейчас у Книжного архитектора нет ни одной авторской настройки: объём сцены и число
глав считаются только формулой, доля сцена/секвель и temperature зашиты в код.

**Не путать роли:** в `state.agents[]` уже есть запись `role:'architect'` — это
«Архитектор сцены», ДРУГАЯ, оперативная роль (планирует якоря/запреты для одной
сцены перед Прозаиком, `ondemand.js`/`agents.js`). Книжный архитектор
(`architect-book.js`, строит скелет глав/сцен на стадии Структуры) сейчас в
`state.agents[]` вообще не представлен.

### 12.1 Три новых поля в Концепции (рядом с `targetWords`)
- **`project.sceneWords`** (число, 0/пусто = авто) — целевой объём одной сцены.
  Автоформула (`clamp(round(totalWords/60), 700, 2000)`) не меняется как дефолт при
  пустом поле. Если автор задаёт явное значение — диапазон шире (300–4000): осознанный
  выбор автора не нужно зажимать той же вилкой, что защищает только автоформулу от
  вырожденных случаев общего объёма.
- **`project.chapterCount`** (число, 0/пусто = «авто») — предзаполняет уже
  существующее поле `#chCount` на стадии Структура (`ui/stages.js:661`, сейчас всегда
  пустое с placeholder «авто»); точечный оверрайд на самой Структуре остаётся рабочим
  поверх этого дефолта, ничего не убираем.
- **`project.pacing`** (`'action'|'balanced'|'reflective'`, дефолт `'balanced'`) —
  select с русскими подписями «Динамичный / Сбалансированный / Медитативный».
  Подменяет одну фразу в `sys` промпта `bookArchitectMessages()`:
  `action` → «обычно 10-20% сцен — sequel», `balanced` → «20-35%» (текущий текст,
  не меняется), `reflective` → «35-50%».

### 12.2 Креативность — новая запись в `state.agents[]`, без новой UI-механики
Добавляется `{ id:'bookArchitect', name:'Книжный архитектор', icon:'🏗', temp:0.6,
enabled:true, role:'bookArchitect' }` в `defaultState()`. Она автоматически
появляется в уже существующей панели «◈ Агенты» (`renderAgentPipeline`,
`ui/diagnostics.js:384`) — генерик-рендер по `state.agents[]`, ничего нового не
пишем в UI. `runBookArchitect()` читает `ag(state,'bookArchitect').temp ?? 0.6`
вместо хардкода `temperature:0.6` (паттерн `ag()` — `pipeline.js:62`).

**Важная оговорка:** базовый `paramSpecs()` (`ui/diagnostics.js:23-27`) сейчас
безусловно добавляет слайдер «Макс. токенов» (диапазон 200–4000) для ЛЮБОГО агента.
Для Книжного архитектора это неверно и опасно — реальный потолок считается
динамически (`archMaxTokens`, 4000–16000 в зависимости от числа сцен) и это
осознанная защита от обрезки JSON-скелета книги; узкий ручной слайдер 200–4000
сломает генерацию длинных книг. Нужно явно исключить `maxTokens` из `paramSpecs()`
для `a.role==='bookArchitect'` (аналогично тому, как уже условно добавляются
`evaluatorThreshold`/`strictness`/`retries` для других ролей) — показываем только
«Температура», авто-расчёт токенов остаётся полностью автоматическим и нечитаемым
из agents[].

## 13. Проверка (live-тест)

По установленной в проекте практике — после реализации прогнать через
`mcp__Claude_Preview__*` на реальном запущенном сервере:
1. Выбор жанра «Фэнтези» в Концепции → стадия «Мир» появляется в рейле + видна
   инлайн-подсказка.
2. «Предложить мир» без затравки → приходят категоризированные факты (реальный вызов
   LLM, не мок).
3. Одобрить часть фактов → появляются в `state.bible[]` с `source:'world'`, видны в
   панели «Память».
4. Перейти в «Структуру», сгенерировать скелет → проверить, что world-факты попали в
   промпт Архитектора (через диагностику/логи вызова).
5. Добавить новый world-факт после того как структура уже построена → баннер
   устаревания структуры появляется.
6. Сгенерировать карту на стадии «Мир» (реальный вызов image-API) → появляется и там,
   и в общей галерее «Иллюстраций».
7. Жанр «Альтернативная история»: пропустить точку развилки → мягкое предупреждение,
   не блокирует; заполнить POD → факт сохраняется, доступен Архитектору.
8. Задать `sceneWords`/`chapterCount`/`pacing` в Концепции → сгенерировать скелет →
   проверить реальные `targetWords` сцен и текст промпта на нужную формулировку темпа.
9. Открыть панель «Агенты» → убедиться, что появилась карточка «Книжный архитектор»
   с одним слайдером «Температура», без слайдера «Макс. токенов».
10. Изменить температуру Архитектора → сгенерировать скелет → подтвердить, что вызов
    LLM ушёл с этим значением (не хардкод 0.6).

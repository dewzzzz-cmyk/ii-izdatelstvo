# Литсовет — Design Spec

**Дата:** 2026-06-17
**Статус:** утверждён к планированию
**Тип:** новый продукт, переиспользующий модули из ИИ-Издательства

---

## 1. Что это

**Литсовет** — инструмент для написания художественных книг, где автор выступает
«режиссёром»: задаёт бриф каждой сцены (2–5 предложений), а цепочка
специализированных ИИ-агентов пишет прозу, сохраняя голос автора и непротиворечивость
мира. В отличие от ИИ-Издательства (визуальный pipeline-редактор на канвасе),
Литсовет — это линейный «письменный стол» из 5 производственных стадий.

**Главное обещание:** управляемая генерация книги, которая (а) звучит голосом автора,
(б) не противоречит сама себе через десятки сцен и несколько книг серии, (в) оставляет
авторский контроль на входе и в середине процесса, а не только на выходе.

**Целевой пользователь:** серийный автор (самиздат / KDP-уровень), которому важны
скорость выпуска и консистентность канона через 5–10 книг.

---

## 2. Цели и не-цели

### Цели
- Полный цикл: Концепция → Голос → Структура → Написание → Редактура.
- Сохранение голоса автора через few-shot примеры (не числовые метрики).
- Непротиворечивость через три Стража (голос / логика / события) и Bible.
- Масштабирование контекста через иерархический Суммаризатор (сцена→глава→книга→серия).
- Режим продолжения серии: загрузка готовых книг → авто-извлечение мира и персонажей.
- Авторский контроль: прогрессивный онбординг, инлайн-директивы агентам,
  редакторский стоп с узлом «рука автора».
- **Два режима работы** (выбор при создании проекта): `режиссёр` (качество — «рука
  автора» и редакторские стопы обязательны) и `фабрика` (скорость — они опциональны,
  возможна пакетная генерация). Режим меняет модель контроля, а не агентов.
- **Диагностический режим** — встроенная наблюдаемость: включать/отключать агентов
  по одному, изолировать на каком этапе возникает проблема, прогон ядра отдельно.
- Экспорт: `.md`, `.docx`, `.epub`.

### Не-цели (YAGNI)
- Совместное редактирование в реальном времени (один автор, локальный инструмент).
- Облачный бэкенд с аккаунтами (локальный сервер-прокси, как в ИИ-Издательстве).
- Иллюстрации / вёрстка обложек / арт-агенты.
- Мобильный клиент.
- Маркетплейс шаблонов жанров.

## 3. Архитектура

### 3.1 Клиент-серверный раздел (как в ИИ-Издательстве)
- **`server.js`** — raw Node.js HTTP-сервер, без зависимостей. Переиспользуется
  как есть. Эндпоинты: `POST /api/generate` (стрим-прокси к любому
  OpenAI-совместимому LLM), `POST /api/backup`, `GET /api/backups`. Опциональный
  `PROXY_TOKEN`. Стрим `choices[0].delta.content` как `text/plain`, ретраи на 429/5xx.
- **Фронтенд** — vanilla JS, без фреймворка и бандлера. Новый, не наследует
  канвас-код ИИ-Издательства.

### 3.2 Переиспользуемые модули из ИИ-Издательства
Переносятся проверенные функции (ссылки — текущие позиции в `app.js`):

| Модуль | Источник | Назначение в Литсовете |
|---|---|---|
| Стрим-прокси | `server.js:269` | без изменений |
| Волновой исполнитель | `runPipeline` `app.js:2726`, `runnableNodes:2710`, `runNode:2273` | оркестрация агентов сцены |
| Bible TF-IDF | `tokensOf:869`, `tfvec:870`, `cosine:871`, `rebuildBibleVecs:872`, `bibleFor:873`, `STOP_RU:868` | семантический поиск канона |
| smartTrunc | `smartTrunc:851` | усечение живого контекста |
| Кэш по хэшу | (в `runNode`) | instant cache hit на неизменных вызовах |
| ZipBuilder + crc32 | `ZipBuilder:1241`, `crc32:1238` | экспорт EPUB |
| typo / md2xhtml | `typo:4150`, `md2xhtml:1267` | типографика и XHTML для экспорта |
| PRICES / оценка стоимости | `PRICES:41` | трекинг бюджета |

### 3.3 Ключевое архитектурное отличие: хранилище
**IndexedDB вместо localStorage.** Серия из нескольких книг + сводки + Bible + выходы
агентов превышает квоту localStorage (~5–10 МБ) и приводит к внезапной потере данных.
Хранилище: IndexedDB с версионированием схемы и ручным экспортом-чекпоинтом (полный
проект в `.json`). API-ключ живёт **только в памяти** (`state.global.apiKey`), никогда
не пишется на диск; `safeReplacer` вычищает `apiKey`/`proxyToken` из экспортов.

## 4. Модель состояния

Единый объект `state`, персистентный в IndexedDB.

| Поле | Описание |
|---|---|
| `project` | название, жанр, поджанр, аудитория, эпоха, синопсис, объём, тип (одиночная / серия), **режим работы (`режиссёр` / `фабрика`)** |
| `style` | стилевые ориентиры (авторы), ползунки (плотность/диалог/темп), запреты, фильтр мата (выкл/мягкий/умеренный/строгий) |
| `voice` | отпечаток голоса: 5+ примеров-предложений автора (единственный управляющий вход в промпт агентов), числовые метрики (только индикатор в UI, НЕ инжектируются в промпт), diff-эволюция между книгами |
| `structure[]` | плоский массив узлов с типами `chapter` \| `scene`; у сцены — бриф, эмоция читателя, целевой объём, статус, текст, версии |
| `bible[]` | записи `keys \| text` + `_vec` (TF-IDF, не сериализуется) |
| `characters[]` | имя, описание, состояние (обновляется Суммаризатором), книга-источник |
| `memory` | иерархические сводки: `scenes{}`, `chapters{}`, `books{}` |
| `series[]` | загруженные книги серии (метаданные + сводки, не полный текст) |
| `agents[]` | конфигурация пайплайна: порядок, параллельные группы, петли, температуры |
| `global` | API URL, модель, ключ (только память), бюджет, ретраи, fallbackURL |
| `log[]` | история выполнения (cap 200) |

## 5. Производственные стадии (UI)

Верхний рейл: **Концепция → Голос → Структура → Написание → Редактура**.
3-колоночный layout с изменяемыми границами (drag dividers): левая (структура + пайплайн),
центр (редактор сцены + бриф + инлайн-директива), правая (анализ сцены + план проекта).

### 5.1 Концепция (прогрессивный онбординг)
Один обязательный вопрос: «О чём ваша книга?». ИИ определяет жанр/тип/аудиторию/эпоху
из описания. Остальное (стилевые ориентиры, ползунки, объём, фильтр мата) — за
сворачиваемыми «Дополнительными настройками». Цель: первое слово книги за < 60 секунд.
Все параметры редактируемы в любой момент.

### 5.2 Голос
Два режима:
- **Образец текста** — автор вставляет 3–5 абзацев прозы → извлечение отпечатка.
- **Мои книги серии** — загрузка `.txt`/`.docx`/`.epub` готовых книг → авто-извлечение
  Bible, персонажей (с состоянием на конец последней книги), diff-эволюции голоса.
  **Объём:** парсинг `.docx`/`.epub` (распаковка ZIP, извлечение XHTML/текста) — это
  **новый код**, не переиспользование. ИИ-Издательство умеет только собирать EPUB
  (`ZipBuilder`), но не разбирать. `.txt` — тривиально. Оценка под-проекта 3 должна это
  учитывать.

Отпечаток передаётся агентам **примерами предложений** (few-shot), числовые метрики
— только индикатор в UI.

### 5.3 Структура
**Книжный архитектор** (один запуск, temp 0.6) генерирует скелет: главы → сцены, у
каждой сцены название, бриф, эмоция читателя, целевой объём, позиция в нарративной арке
(завязка/развитие/кульминация/развязка). Автор редактирует любую сцену до начала
написания.

### 5.4 Написание
Основной цикл. Подробно — раздел 6.

### 5.5 Редактура
Финальная вычитка по главам, экспорт.

## 6. Агентный пайплайн (стадия Написание)

Волновой исполнитель (переиспользован). Порядок и параллельные группы настраиваются в
«Режиме эксперта» (drag-and-drop скрыт для новичков).

```
Архитектор сцены (0.4)
        │  JSON: детали, якоря, запрещённые слова — НЕ пишет прозу
        ▼
Прозаик (0.85)  ◀──┐
        │          │ возврат с замечаниями
        ▼          │
Оценщик (0.2) ─────┘  отдельный агент; НЕ видит контекст генерации;
        │              балл 1–10 + замечания; выход при балле ≥ порог
        │              (макс. итераций → стоп с лучшим вариантом)
        ▼
   ┌────────────────── параллельный шаг ──────────────────┐
Страж голоса (0.2)   Страж логики (0.2)   Страж событий (0.2)
   стиль/ритм          причинность/         кто что знает/
   (цитаты образца)    время/пространство   состояния персонажей
   └──────────────────────────────────────────────────────┘
        │  все три только флагуют, не переписывают
        ▼
Линейный редактор (0.3) — опционально: убирает ярлыки, варьирует ритм
        ▼
Суммаризатор (авто, после одобрения) → сжимает сцену в ~60 слов,
                                        обновляет состояния персонажей
```

**Агенты и их роли:**

| Агент | Temp | Видит | Возвращает |
|---|---|---|---|
| Книжный архитектор | 0.6 | жанр, синопсис, объём, голос | скелет книги |
| Архитектор сцены | 0.4 | бриф, Bible, сводки | JSON якорей/запретов |
| Прозаик | 0.85 | всё (раздел 7) | 500–800 слов прозы |
| **Оценщик** | 0.2 | **только черновик + критерии + бриф** (не рассуждения Прозаика) | балл + замечания |
| Страж голоса | 0.2 | черновик + примеры голоса | флаги + цитаты образца |
| Страж логики | 0.2 | **только факты** (сводки + Bible + состояния) | флаги противоречий |
| Страж событий | 0.2 | **только факты** (хронология + состояния) | флаги знания/эмоций |
| Линейный редактор | 0.3 | черновик + запреты | отредактированный текст |
| Суммаризатор | 0.3 | одобренная сцена/глава/книга | сводка + обновл. состояний |

**Принцип разделения:** Стражи логики/событий и Оценщик НЕ видят инструкции стиля —
чтобы не смешивать задачи. Оценщик независим от генератора (устранение
self-consistency bias).

**Каждый агент в пайплайне включаем/отключаем** (диагностический режим, раздел 12).
Это не только наблюдаемость — это способ проверить спорные решения на реальной прозе
вместо споров в дизайне:

| Спорное решение (ревью) | Как проверяем диагностическим режимом |
|---|---|
| Архитектор сцены (отдельный JSON-вызов) душит литературность? | прогон с ним и без него на одной сцене → сравнить прозу |
| Независимый Оценщик-судья реально поднимает качество или сходится к «гладкой ИИшной прозе»? | прогон с петлёй и без → человеческий замер; альтернатива баллу — режим парного сравнения (черновик vs предыдущая одобренная сцена) |
| Три Стража или один объединённый фактологический? | Стражи логики+событий можно слить в один вызов и сравнить |
| 5 few-shot примеров удерживают голос на дистанции? | замер дрейфа через 10–20 сцен; опция — ротация 8–12 примеров |

Эти альтернативы — настраиваемые варианты, а не переписывание. Решение, какой состав
оставить по умолчанию, принимается по данным первых прогонов, а не заранее.

**Петли:** каждый агент может иметь петлю с условием выхода (балл ≥ N / объём ≥ N /
ошибок = 0 / кастом) и лимитом итераций. Оценку в петле всегда даёт независимый агент,
не сам генератор. **Дефолты:** порог Оценщика = 7 (из 10), макс. итераций = 3.
Значения настраиваемы per-agent; при достижении лимита — стоп с лучшим по баллу
вариантом.

### 6.1 Рубрика Оценщика и защита от «гладкой ИИшной прозы»

Главный риск (все эксперты): без явных критериев петля сходится к обтекаемому
median-тексту, который нравится модели-судье, но мёртв для читателя. Оценщик не ставит
голый балл — он оценивает по **рубрике из 5 осей** (каждая 1–10):

| Ось | Что измеряет | Анти-паттерн (штраф) |
|---|---|---|
| Свежесть образа | небанальные метафоры/детали | клише, штампы, «дежурные» образы |
| Ритм | вариативность длины предложений | монотонность, всё одной длины |
| Конкретность | предметность вместо абстракций | общие слова, «эмоц. ярлыки» |
| Голос | совпадение с few-shot образцом | регрессия к нейтральному стилю |
| Соответствие брифу | сюжет + эмоция читателя достигнуты | уход от задачи сцены |

Порог = средневзвешенное ≥ 7 **и** ни одна ось < 5. «Антигладкость» (свежесть + ритм)
— явные оси, чтобы петля **ловила** дрейф к гладкости, а не полировала его.

**Человек в контуре (калибровка).** LLM-судья сам страдает дефектом вкуса. Поэтому:
периодически (раз в главу или каждые N сцен) автору предлагается **blind-rating** —
оценить 2–3 выхода вслепую. Эти оценки калибруют порог Оценщика под реальный вкус
автора, а не оставляют «7 из 10» висеть без шкалы. В режиме `Фабрика` калибровка реже и
опциональна, в `Режиссёр` — регулярна. Это и есть обязательное человеческое звено,
которого требовало ревью: инструмент не только показывает дрейф, но и привязывает
автоматическую оценку к человеку.

**Примечание по диаграмме:** диаграмма выше показывает только стадию Написание и
начинается с Архитектора сцены. Книжный архитектор (раздел 5.3) принадлежит стадии
Структура и запускается один раз до Написания — в этой диаграмме он не показан намеренно.

## 7. Контекст агента (иерархическая память)

Чтобы не терять понимание книги через десятки сцен и серию книг, контекст собирается
послойно. Суммаризатор сжимает завершённое после одобрения:

| Уровень | Размер | Кто создаёт | Когда |
|---|---|---|---|
| Сцена | ~60 слов | Сценный суммаризатор | после одобрения сцены |
| Глава | ~200 слов | Главный суммаризатор | после редакторского стопа главы |
| Книга | ~300 слов | Книжный суммаризатор | после последней сцены книги |
| Серия | ~300 слов/книга | (из загруженных книг) | при импорте |

**Что получает Прозаик на сцену:**
1. Голос — 5 примеров-предложений + запреты (~270 слов)
2. Серия + книга — сводки прошлых книг и завершённых глав (сжато)
3. Текущая глава + персонажи — мини-сводки сцен + актуальные состояния (~400 слов)
4. Bible — топ-5 TF-IDF матчей по брифу (~500 слов)
5. Живой контекст — полный текст предыдущей сцены + бриф + эмоция + выход Архитектора (~1000 слов)

**Бюджет:** серия 5 книг, глава 10 ≈ 5 750 слов ≈ 8 000 токенов (< 5% от 200K).
Масштабируется до 50+ книг. Живой контекст усекается через `smartTrunc` (первые/последние
45%) при превышении.

### 7.1 Алгоритм сборки (псевдокод)

Сборка — новый код, не перенос. `buildSceneContext(scene)`:

```
1. layers = []
2. layers.push( voice.examples[0..4] + style.forbidden )        // фикс ~270 сл
3. if project.type == 'серия':
      layers.push( series.map(b => b.summary) )                  // ~300 сл/книга
4. layers.push( book.chapters.filter(done).map(c => c.summary) ) // ~200 сл/глава
5. layers.push( currentChapter.scenes.filter(done).map(s => s.summary) ) // ~60 сл
6. layers.push( serializeCharacterStates(scene.presentChars) )   // имя: состояние
7. layers.push( bibleFor(scene.brief).slice(0, 5) )              // TF-IDF топ-5
8. live = prevScene.fullText + scene.brief + scene.emotion + architectOutput
9. layers.push( live )
10. assembled = join(layers, SEP)
11. if tokensOf(assembled) > BUDGET:
       // усекаем в порядке приоритета (НЕ голос, НЕ живой контекст):
       // сначала сводки серии → главы → сцены, в последнюю очередь
       // живой контекст усекается через smartTrunc (45/45)
       assembled = truncateByPriority(layers, BUDGET)
12. return assembled
```

**Единицы бюджета.** `BUDGET` задаётся в **токенах** (дефолт 12 000 на сборку, ~6% от
200K — запас на ответ модели). Слова в разделе 7 — ориентир для оценки; фактический
учёт и усечение — в токенах (оценка как в ИИ-Издательстве: кириллица ≈ длина/2).

**`truncateByPriority(layers, BUDGET)`:**
```
order = [серия, главы, сцены]   // режем в этом порядке, остальное не трогаем
for layer in order:
    while tokens(all) > BUDGET and layer not empty:
        drop oldest entry from layer       // самые старые сводки первыми
if tokens(all) > BUDGET:                    // редкий край: голос+Bible+живой > BUDGET
    live = smartTrunc(live, remaining)      // ужимаем живой контекст 45/45
// голос и запреты НЕ режутся никогда — без них агент теряет задачу
```

**`scene.presentChars`** — заполняется Архитектором сцены (он и так разбирает, кто в
сцене) ИЛИ берётся из брифа автором; до первой генерации — из скелета Книжного
архитектора. Сериализация: `serializeCharacterStates` (новый) → `«Имя — состояние»`,
по строке на персонажа.

**Прочие изменения в переиспользуемом коде:**
- `bibleFor` сейчас сканирует `state.bible` по тексту контекста — переучить искать по
  `scene.brief` (один аргумент, тот же TF-IDF/cosine; сигнатура уже `bibleFor(text)`).
- `SEP` — явные разделители-заголовки слоёв (`=== ГОЛОС ===` и т.п.), чтобы модель
  различала уровни.
- Политика усечения — отдельная чистая функция (тестируется на переполнении).

**Риск дрейфа:** сжатие с потерями — ошибка в сводке главы каскадно отравляет книгу.
Дрейф **численно** = падение оси «Голос» Оценщика (раздел 6.1) ниже скользящего среднего
по предыдущим сценам на ≥ 2 пункта, ИЛИ рост cosine-расстояния новой сцены до образца
голоса выше порога. При срабатывании — флаг в диагностике с указанием подозрительной
сводки.

Митигации: (1) сводки видимы автору (вкладка «Память») и редактируемы; (2) **версионирование
памяти** — каждая сводка хранит предыдущие версии, откат при обнаружении отравления
(UI: список версий сводки, кнопка «откатить»; гранулярность — на уровне записи сводки);
(3) диагностический режим показывает, какие слои попали в промпт конкретной сцены —
чтобы отследить источник дрейфа. Замечание ревью справедливо: автор «за 60 секунд» не
будет вычитывать сводки сам, поэтому отлов дрейфа опирается на численный сигнал + (2) и
(3), а ручная правка — резервный путь.

## 8. Авторский контроль

**Режим работы** (выбор при создании, переключаем позже):
- **`Режиссёр`** — «рука автора» (раздел ниже) и редакторские стопы **обязательны**,
  качество приоритетно. Дефолт; совпадает с исходным запросом «произведение литературы».
- **`Фабрика`** — стопы и «рука автора» **опциональны**, возможна пакетная генерация
  нескольких сцен подряд, скорость приоритетна. Для серийных KDP-авторов.

Режим влияет только на модель контроля (обязательность узлов), не на состав агентов.

- **Прогрессивный онбординг** — один вопрос на входе (раздел 5.1).
- **Бриф сцены** — вкладки «Сюжет» (что происходит) и «Эмоция читателя» (что чувствует
  читатель в финале, отдельно от сюжета).
- **Инлайн-директива** — поле под текстом сцены: «скажи агенту что изменить» + быстрые
  чипы. Флаги Стражей имеют кнопку «→ Дать директиву Прозаику».
- **Редакторский стоп** — при завершении главы следующая блокируется до прочтения.
- **Узел «рука автора»** — стоп требует не «прочитал», а минимум один абзац,
  переписанный автором своими словами. Без этого следующая глава заблокирована.
- **Выделение текста → плавающее меню** — Переписать / Сократить / Усилить / Деталь мира
  / Продолжить.

## 9. Правая панель: анализ + план проекта

- **Верх — Анализ сцены:** флаги трёх Стражей с цитатами-источниками, детали сцены,
  голос (5 примеров). Единый счётчик-приоритет: «5 проблем, 1 критичная» — чтобы три
  потока флагов не перегружали.
- **Низ — План проекта (роадмап):** этапы производства, прогресс глав, цели (объём/
  бюджет), **контекст для агентов** (теги: установлено в мире / тон / запрещено /
  следующая сцена). Вкладка «Память» — редактируемые сводки.

## 10. Экспорт

## 11. Обработка ошибок

- Ошибка агента не останавливает пайплайн: узел получает `status='error'`, нижестоящие
  продолжают с `[АГЕНТ ПРОВАЛИЛСЯ]` (как в ИИ-Издательстве).
- HTTP 502/503 → ретрай на `fallbackURL` если задан.
- **JSON-схема Архитектора:** выход Книжного архитектора и Архитектора сцены
  валидируется по схеме с ретраем — кривой JSON ломает весь пайплайн вниз.
- IndexedDB: при превышении квоты — мягкое предупреждение + предложение экспорта-чекпоинта,
  не падение.

## 12. Тестирование и диагностический режим

### 12.1 Автотесты
- **Чистые функции** (TF-IDF, smartTrunc, сборка контекста, политика усечения, валидатор
  схемы, diff голоса) — юнит-тесты на входах/выходах.
- **Сборка контекста** — тест: для сцены N в книге M контекст содержит ожидаемые слои,
  в правильном порядке, и не превышает бюджет; при переполнении усекается по приоритету.
- **Стражи** — тесты на заготовленных противоречиях (запертая дверь, имя ниоткуда,
  хронология) → ожидаемые флаги.
- **Экспорт EPUB** — тест валидности ZIP/структуры (как в ИИ-Издательстве).
- **End-to-end** — прогон одной сцены через весь пайплайн на mock-LLM (детерминированные
  ответы) → проверка состояния и памяти.

### 12.2 Диагностический режим (встроенный)
Не тестовый стенд, а постоянная функция продукта — отвечает на тревогу ревью «проверить
прозу рано» через наблюдаемость, не ломая полный порядок сборки.

- **Toggle агентов** — каждый агент включается/отключается. Можно прогнать голое ядро
  (Прозаик в одиночку), затем навешивать Оценщик, Стражей, Линейного редактора по одному
  и видеть, что меняется в прозе.
- **Трейс прогона** — по `run_id` сохраняется: какие слои контекста ушли в промпт каждого
  агента, его вход/выход, токены, стоимость, итерации петли, вердикты Оценщика и Стражей.
- **Изоляция этапа** — при плохом результате видно, на каком агенте проза «сломалась»:
  слабый бриф Архитектора? петля Оценщика загладила? Страж зарезал живое место?
- **A/B одной сцены** — прогнать одну сцену двумя конфигурациями (с Архитектором / без;
  петля / без петли; 3 Стража / 1) → сравнить выходы рядом.

Диагностический харнесс строится в под-проекте 1 (каркас) и протягивается через все
последующие — каждый новый агент сразу попадает под наблюдение.

## 13. Декомпозиция реализации (для writing-plans)

Продукт большой; план разбивается на под-проекты, каждый — рабочий и тестируемый.
**Порядок переставлен** (по ревью): главная гипотеза — «проза вообще читаема и звучит
голосом автора» — проверяется на под-проекте 2, а не на 60% пути.

1. **Каркас + хранилище + диагностический харнесс** — IndexedDB-слой, модель `state`,
   экспорт-чекпоинт, перенос server.js, 3-колоночный layout с рейлом и изменяемыми
   границами, переключатель режима Режиссёр/Фабрика, **трейс прогонов по `run_id` и
   toggle агентов** (раздел 12.2) — закладывается здесь, протягивается дальше.
   **Критерий приёмки:** через голое ядро (один заглушечный агент) проходит и
   сохраняется одна реальная сцена, трейс виден в диагностике.

> **Правило модульности фронтенда.** Текущий `app.js` ИИ-Издательства — монолит 4000+
> строк. Чтобы новый фронтенд не повторил это: разбивка на ES-модули по стадиям и
> ответственностям (хранилище, сборка контекста, оркестрация агентов, каждая стадия UI),
> бюджет строк на файл. Оркестрацию пайплайна писать линейно (петлевую механику и кэш
> переиспользовать из волнового исполнителя как **идеи**, не тащить графовый код целиком —
> см. замечание ревью о натяжке).

2. **Голое ядро прозы (вертикальный срез)** — Голос (только режим образца) + Прозаик +
   Оценщик с рубрикой (раздел 6.1) на **минимальном руками-собранном контексте**, одна
   сцена. **Критерий приёмки и цель всего этапа:** автор читает реальную прозу своим
   голосом и судит — пишет ли продукт хорошо. Здесь же первая калибровка blind-rating.
   Это та самая ранняя проверка, ради которой переставлен порядок.
3. **Bible + иерархическая память** — перенос TF-IDF, Суммаризатор, полная сборка
   контекста (раздел 7.1, заменяет ручную из ПП-2), численный детектор дрейфа, вкладка
   «Память» с откатом версий.
4. **Структура** — Книжный архитектор, скелет, редактор сцен, валидатор JSON-схемы.
5. **Полный пайплайн** — Архитектор сцены, три Стража, Линейный редактор, петли и
   параллельные группы, режим эксперта. Спорные агенты включаются и проверяются
   диагностикой (раздел 6).
6. **Авторский контроль** — онбординг, бриф с эмоцией, инлайн-директивы, плавающее меню,
   редакторский стоп + узел «рука автора» (обязательность по режиму).
7. **Правая панель** — анализ сцены, единый счётчик флагов, роадмап, контекст агентов.
8. **Режим серии** — импорт `.txt`/`.docx`/`.epub` (новый парсинг — см. 5.2),
   авто-извлечение Bible/персонажей, diff-эволюция голоса.
9. **Редактура + экспорт** — финальная вычитка, `.md`/`.docx`/`.epub`/`.json`.

Каждый под-проект получает свой spec → plan → реализацию. Этот документ — зонтичный
дизайн; первыми в разработку идут под-проекты 1 и 2 (каркас + голое ядро прозы).

## 14. Безопасность

- API-ключ только в памяти, никогда на диск.
- `safeReplacer` вычищает `apiKey`, `apiKeys`, `proxyToken` из любых экспортов.
- Загруженные книги серии и образцы текста обрабатываются локально, не уходят никуда
  кроме выбранного пользователем LLM-эндпоинта.
- Контент-фильтр (мат) — настройка проекта, инжектируется в системный промпт агентов.

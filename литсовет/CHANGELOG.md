# Changelog

Версия видна в приложении (шапка и «Настройки») и через `GET /api/version`.
Бампается при каждом серьёзном фиксе пайплайна/сервера — не при мелкой
правке промпта или косметике UI.

## 1.15.0 — 2026-07-23

Разбор статистики 44 реальных вердиктов Оценщика показал, что 3 из 6 осей
рубрики (свежесть, конкретика, темп) практически заморожены на средних
значениях (LLM-as-judge central-tendency bias при отсутствии эталона для
сравнения) — арифметический потолок «личного лучшего» по каждой оси даёт
7.29, что ниже порога принятия 7.5. Плюс кросс-сценовые проблемы (повтор
одного и того же комедийного приёма, пассивность героя, однообразная подача
экспозиции), которые не видны ни одному агенту, читающему только одну сцену
за раз. Все изменения сделаны жанронезависимыми — инструмент не подгоняется
под конкретную книгу.

- **Калибровка Оценщика по осям без внешнего эталона.** Freshness/Concrete/
  Pace получили few-shot примеры (3/6/9, жанронейтральные иллюстрации) и
  правило «мост между цитатами и баллом» (0 цитируемых проблем → ≥6, 2-3 →
  ≤6; балл 5 как ленивый дефолт запрещён явно). Freshness видит уже
  использованные в этой книге клише (`usedCliches`), Pace сравнивается с
  медианой длины уже принятых сцен этой же книги (самореференция, не
  жанровый норматив).
- **Отслеживание пассивности героя** — переиспользует существующий вопрос
  Читателя-стража («герой пассивен?») без нового вызова LLM; систематическая
  пассивность (≥3 из последних 6 сцен) всплывает в отчёте Критика.
- **Новый детектор кросс-сценовых приёмов** (`src/craftsignals.js`) —
  извлекает структуру комедийных/напряжённых битов сцены (закрытые категории
  setup/style) и СВОБОДНУЮ метку источника (`channel`), которую называет сама
  модель под конкретную книгу — не завязано на фиксированный список тропов
  вроде «ИИ-спутник», работает для любого жанра/приёма. Группировка по
  нечёткому сходству меток (TF-IDF/cosine, как в bible.js). Отчёт Критика
  показывает: повторяющийся приём (≥4 раз), доминирование одного источника
  экспозиции (≥60% сцен с экспозицией).

## 1.14.0 — 2026-07-23

Найдено и починено при полном редакторском чтении книги (обрыв сцен на
полуслове) и последующем разборе, почему целевая правка не поднимала балл.

- **Обрыв стрима токенами/сетью маскировался под штатное завершение.**
  `server.js` глотал ошибку оборванного апстрим-соединения пустым `catch{}`
  — клиент получал формально успешный ответ и не мог отличить обрыв от
  нормального конца. Теперь сервер помечает обрыв маркером в теле ответа,
  `llm.js` ловит его и ретраит как обычную сетевую ошибку.
- **`looksTokenTruncated` не была подключена к ветке правки черновика** —
  самому частому пути сцены по пайплайну. Теперь обрыв не может победить в
  отборе `best` ни в одной ветке (с/без Оценщика, авто/эскалация).
- **Токенные бюджеты прозы были занижены** относительно реальной плотности
  (~2.78 ток/слово по факту против заложенных 2.5). Прозаик/Линейный
  редактор: 2.5→3.5 ток/слово, потолок правки 5000→11000, потолок ретрая
  первого черновика 8000→16000.
- **`opts.directive` (целевая правка по конкретным находкам) терялась после
  первой итерации** — в конце каждой итерации директива полностью
  перезаписывалась текстом, собранным заново только из ТЕКУЩИХ находок
  Стражей/Оценщика. Теперь «стоячее» указание домешивается в директиву на
  каждой итерации, а не только на первой.
- **Потолок правки масштабировался только от длины прозы**, не от объёма
  критики — теперь растёт вместе с длиной директивы (запас под текст
  [РАЗБОР], а не только под переписанную прозу).
- **Реальный `usage` от апстрима теперь долетает до клиента**
  (`stream_options.include_usage`) — токен-учёт и остальные бюджетные
  решения опираются на факт, а не на клиентскую оценку `estimateTokens`.
- **Оценщик реально упирался в лимит токенов** (живой факт: `tokensOut`
  ровно на заявленных 1950 → JSON обрывался → вердикт не парсился). Лимит
  поднят (1950→3200), добавлен один ретрай с удвоенным лимитом при провале
  парсинга.
- **Нераспарсенный вердикт Оценщика мог выиграть отбор `best`** только за
  счёт отсутствия критических флагов Стражей — сцена уходила в книгу без
  реальной оценки качества (`lastEval.ok:false`), а не с низкой оценкой.
  Теперь `verdict.ok` — обязательное условие «чистого» черновика.
- Добавлен `GET /api/version` — сверка задеплоенной версии без открытия UI.
